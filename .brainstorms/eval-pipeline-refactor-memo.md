# Decision Memo — EVAL Pipeline Refactor

## What this is

Architecture refactor of the skill evaluation pipeline. Not a single new skill — a restructuring that introduces two new repo concepts (agent definitions, protocols), demotes two skills to agents, creates one new agent, adds a trace protocol, and modifies three existing skills.

## Problem statement

1. **EVAL.md is missing execution criteria.** Skills with multi-phase workflows, subagent dispatch, and model selection have no way to evaluate whether the skill followed its prescribed execution pattern. Only output quality (EVAL-O) and structural conformance (EVAL-S) are graded today.

2. **Too many skills.** `skill-eval-judge` and `skill-eval-prompter` are full skills (SKILL.md, EVAL.md, registry entry, symlink) but are never user-invoked directly. They're internal recipes dispatched by skill-creator and write-skill-eval. This adds token noise and maintenance overhead.

3. **No execution observability.** Claude Code has no built-in trace collector (unlike LangGraph+Langfuse). When a skill dispatches subagents with wrong models or skips required phases, there's no way to detect it during evaluation or debugging.

## Key decisions

### 1. Three-tier taxonomy: Skills, Agents, Protocols

| Category | What it is | Where it lives | User-invocable? |
|----------|-----------|----------------|-----------------|
| **Skill** | User-facing recipe with lifecycle (EVAL.md, improvement loop) | `src/<domain>/<skill>/SKILL.md` | Yes |
| **Agent** | Internal recipe dispatched by skills | `src/agents/<name>.md` | No — dispatched via `Agent()` |
| **Protocol** | Shared convention/schema injected into agents when needed | `src/protocols/<category>/<name>.md` | No — injected by observer |

**Boundary test:** "Would a user ever say this as a command?" If yes → skill. If no → agent definition.

**Metaphor:** Skill = recipe (user-facing). Agent definition = recipe (internal). LLM instance = cook. Subagent = cook delegated by another cook. Protocol = form a cook fills out when asked.

### 2. Agent definitions with symlink sharing

Agent definitions live in `src/agents/` as SSOT. Skills that dispatch them create symlinks in their own `agents/` folder:

```
src/agents/                                    # SSOT
  skill-eval-judge.md
  skill-eval-prompter.md
  skill-eval-auditor.md

src/skills/skill/skill-creator/agents/                # consumer declares dependencies
  skill-eval-judge.md → ../../agents/skill-eval-judge.md
  skill-eval-prompter.md    → ../../agents/skill-eval-prompter.md
  skill-eval-auditor.md → ../../agents/skill-eval-auditor.md

src/skills/skill/write-skill-eval/agents/
  skill-eval-judge.md → ../../agents/skill-eval-judge.md
  skill-eval-prompter.md    → ../../agents/skill-eval-prompter.md
  skill-eval-auditor.md → ../../agents/skill-eval-auditor.md

src/skills/skill/improve-skill/agents/
  skill-eval-judge.md → ../../agents/skill-eval-judge.md
```

Benefits: SSOT, DRY, explicit dependency declaration via `ls agents/`, no registry pollution.

### 3. EVAL-E: Execution criteria tier

New tier in EVAL.md between Structural and Output:

```markdown
## Execution Criteria

- [ ] **EVAL-E01:** Phase 1 subagents dispatch per test file
- [ ] **EVAL-E02:** Phase 1 subagents use model: sonnet
- [ ] **EVAL-E03:** Phase 2 cross-matching runs on model: opus
...
```

Generated by `src/agents/skill-eval-auditor.md`. Dispatched by skill-creator (Phase 5) and write-skill-eval. Not all skills have execution criteria — only those with orchestration patterns (subagent dispatch, model selection, phase gates, shortcuts).

Proof of concept: 5 EVAL-E criteria already added to `write-test-coverage/EVAL.md`.

### 4. Execution trace protocol (observability toggle)

```
src/protocols/traces/execution-trace.md
```

Not baked into every execution. Injected by an observer (improve-skill, auto-research, user debugging) when needed. Zero overhead in normal runs.

The trace is self-reported by the cook — the subagent executing the skill appends a structured trace to its response when asked:

```yaml
## Execution Trace

phases_executed: [0, 1, 2, 3]
test_file_count: 12

agents_dispatched:
  - phase: 1
    purpose: scan test file
    target: tests/test_auth.py
    model: sonnet

workflow_decisions:
  - decision: subagent dispatch strategy
    chose: per-file dispatch
    reason: "12 test files >= 10 threshold"

precondition_checks:
  - check: intent document exists
    result: pass
    path: docs/auth/SPEC.md

context_isolation:
  phase1_prompt_contains_acs: false
```

**Nesting:** configurable — shallow (one level, default) or deep (recursive subagent traces).
**Cost:** no caps — full visibility. Half-visibility defeats the purpose.
**Persistence:** traces written to file for review.

Future trace types: `decision-trace.md` (judgment calls), `context-trace.md` (token budget, files read).

### 5. improve-skill gets EVAL-E support

Currently only grades EVAL-S (structural, static analysis) and EVAL-O (output, behavioral). Needs to:

1. Recognize EVAL-E as a tier in EVAL.md
2. Inject execution trace protocol into behavioral test subagents
3. Grade returned traces against EVAL-E criteria
4. Symlink to `src/agents/skill-eval-judge.md` for criteria regeneration

## Open questions

1. **Trace persistence location.** `research/traces/` per skill? Or a central `traces/` directory? Per-skill keeps traces near the skill; central makes cross-skill analysis easier.
2. **Structural criteria.** Stays hardcoded in improve-skill for now. May become an agent later if per-skill structural checks are needed.
3. **LangGraph migration.** Agent definitions → node prompt templates. Protocols → node middleware. Not blocking — current design is a stepping stone.
4. **Protocol folder structure.** `src/protocols/traces/` and `src/protocols/operations/` sketched but only `traces/execution-trace.md` is in scope for this refactor.

## Execution order

3. Modify skill-creator (agent symlinks + Phase 5 execution criteria)
4. Modify improve-skill (EVAL-E support + trace injection)

6. Cleanup: remove demoted skills, update registry
7. Test: run skill-creator on a skill, verify EVAL-E generated, run improve-skill, verify trace + grading

## Discarded alternatives

- **Expand skill-eval-judge for execution criteria.** Rejected: persona conflict — judge ≠ conformance checker.
- **New standalone skill for skill-eval-auditor.** Rejected: overhead for always-dispatched, never user-invoked workload.
- **Absorb improve-skill into skill-creator.** Rejected: different lifecycles (create once vs improve repeatedly).
- **skill-creator as argument router.** Rejected: already the longest skill, routing logic adds complexity.
- **Trace as permanent agent output.** Rejected: wasteful — toggle instead, zero overhead in normal runs.
- **Trace format inside improve-skill only.** Rejected: not shared — multiple consumers need it.
- **Execution trace as an agent definition.** Rejected: it's not a cook, it's a form. Protocol, not agent.
