# Brainstorm Notes — sync-docs

## Status
Phase: B
Outcome: skill

## Resolved
- Input shape: git diff, triggered post-code-review. Diff is the source of truth for what changed.
- Trigger model: all three are valid — user-invoked, pipeline stage, CLAUDE.md hook. Skill must support headless/autonomous execution with conservative defaults.
- CLAUDE.md as *invocation trigger* (not logic): a lightweight CLAUDE.md rule "always run /sync-docs after any coding change" is valid and doesn't bloat global context — logic stays in the skill.
- Test docs gap: primarily *discovery* (which test plan file to find) + *consistency* (new test cases must follow existing doc's conventions/rules, not just be appended raw)
- Doc scope in: README.md, spec, spec-summary (auto-regenerated from spec if spec changes), engineering guide, test-docs
- Doc scope out: non-doc artifacts (code, config, etc.)
- Spec/test relationship: spec covers *requirements + acceptance criteria*. test-docs covers *implementation-level test cases*. There is no separate "spec for test" — spec's acceptance criteria IS the test requirements layer. write-test-docs is implementation-level and may not always need updating (e.g., refactor with no behavioral change). Sync-docs should NOT try to resolve this architecture — it reflects what exists.
- Change type → spec update rule: spec only updates when a change reveals a missing/wrong requirement (bug that exposes a gap) or adds a genuinely new requirement. Not every change touches spec. But eng-guide almost always gets a small update for any behavioral change.
- Exit condition revised: only truly cosmetic changes (whitespace, comments, formatting with no API/behavior impact) are no-ops. Variable renames CAN affect eng-guide if they're in a public interface. Threshold is behavioral impact, not size of diff.
- Escalation: redirect to the *specific* skill (e.g., /write-engineering-guide, /write-test-docs), not /doc-authoring generically. Skill knows which doc is overwhelmed.
- Missing doc handling: patch-docs delegates creation to the right write-* skill as a subagent, then patches the result. Only for docs directly relevant to the diff — not aspirational creation.
- Section-level reads: patch-docs reads only the relevant doc section (via .doc-map.yaml), NOT the full document. No heuristic staleness detection — unreliable without KG. Once KG exists, staleness is a separate capability (doc-audit or KG query).
- Escalation rule: if patching would require a new top-level section, redirect to the specific write-* skill.
- Quality mechanism: CLAUDE.md invocation trigger ensures it always runs. Skill's own quality is consistency enforcement (new content follows existing doc's format/naming/coverage rules).

## Key Insights
- Gap is real: no skill covers "given this diff, apply minimal targeted doc updates"
- doc-authoring routes to the right doc type but doesn't handle change-driven patching
- write-engineering-guide creates from source code post-implementation; doesn't handle incremental updates
- write-test-docs creates a full test plan; doesn't append to an existing one
- pytest --cov gives line coverage (a number). That doesn't tell a team WHAT is tested or WHAT is missing at the scenario/behavior level. A test coverage register is real team knowledge, not just a metric.
- test-coverage.md warrants its own skill (write-test-coverage), initialized separately, updated incrementally by patch-docs
- Doc-to-code registry (.doc-map.yaml or similar) is the right solution to patch-docs' discovery problem — also serves as the KG foundation the user wants to build. Elegant: patch-docs reads it for O(1) discovery instead of heuristic search every run.
- Config files: only "documentation config" (e.g., .env.example) is in scope. Tool-consumed config (pyproject.toml, docker-compose.yaml) is code and should not be touched by patch-docs.

## Tensions
- Scope ceiling: skill needs a hard rule for when to refuse and redirect to full regen.
- Exit condition: behavioral impact threshold, not diff size.
- test-coverage.md is a separate skill (write-test-coverage or similar) — patch-docs is NOT responsible for initializing it, only for triggering it or making small incremental updates to it.
- Config files (.toml, .yaml): valid target only if they serve as documentation (e.g., .env.example, annotated config docs). NOT if they are tool-consumed artifacts (pyproject.toml, docker-compose.yaml) — those are code, not docs.
- Discovery: doc-to-code registry generated by patch-docs on first run if absent. No separate skill needed — mapping is straightforward (glob for doc files, read first few sections, match to code paths via references/imports). Patch-docs maintains it on subsequent runs.
- Config file detection: need a heuristic for "doc config" vs "tool config." Candidate: check if the file is referenced in a build tool / CI pipeline / package manager → tool config → out of scope.

## Discarded
- Staleness flagging: dropped. Unreliable without KG (only catches things by accident based on what section patch-docs happens to read). Once KG exists, staleness detection becomes a separate capability, not a side effect of patching.
- Full doc reads: unnecessary. Section-level reads via .doc-map.yaml are sufficient and keep cost bounded.
- Fail-loudly on missing docs: replaced by delegate-to-write-skill. Failing adds friction for no benefit when patch-docs already knows which skill to invoke.

## Lens Notes (Phase B)
### Usability
- Trigger phrases clear: "update the docs", "patch docs", "sync docs after this change"
- Name communicates intent
- Output: patch summary showing what was touched and what was deliberately skipped, tied to the changes that drove each update
### Robustness
- Missing docs: delegate to write-* skill, fail loudly if skill not found
- Pre-existing staleness: ignore, only address the diff
- Section-level reads keep scope bounded
- Sibling skill references: fail loudly if missing, no silent degradation
### Maintenance
- Sibling skill names are hardcoded — accepted risk, mitigated by loud failure on missing skill
- .doc-map.yaml format is simple enough to be stable
- No version-specific or API-specific coupling
### Preciseness
- Core injection: change-type classification, section-level targeting via .doc-map.yaml, escalation policy, consistency rule (match existing conventions)
- All four are things Claude does NOT do well unprompted. Tight.
### Boundary
- Patch-docs patches existing sections; write-* skills create/regenerate
- Missing docs: delegate creation then patch. Only for diff-relevant docs.
- Staleness detection: out of scope, future KG/doc-audit concern
- Escalation: new top-level section needed → redirect to specific write-* skill
