# Decision Memo — patch-docs

## What I want

A skill that takes a git diff and applies targeted, minimal updates to existing documentation — without going through the full /doc-authoring chain. Most coding work (refactors, bug fixes, small features) doesn't warrant regenerating docs from scratch, but the docs still need to stay current. This should also handle test coverage docs incrementally, and README.md which always drifts on coding projects.

## Why Claude needs it

Without this skill, Claude either:
- **Forgets to update docs entirely** unless explicitly asked — the most common failure
- **Over-engineers the update** by regenerating whole documents through write-* skills when only a section needs a line added
- **Doesn't know which docs to find** given a code change — discovery is the hard part

No existing skill covers "given this diff, find affected doc sections and patch them":
- `doc-authoring` routes to the right doc type but doesn't do change-driven patching
- `write-engineering-guide` creates from scratch post-implementation
- `write-test-docs` generates a full test plan, not incremental updates

## Injection shape

**Policy + judgment.** The core injections are:

1. **Change-type classification** — diff → feature/refactor/bugfix/cosmetic → which doc types are affected (e.g., bug fix rarely touches spec, almost always touches eng-guide)
2. **Section-level targeting** — use `.doc-map.yaml` to read only the relevant section, not the full document
3. **Escalation policy** — if patching would require a new top-level section, delegate to the specific write-* skill
4. **Consistency enforcement** — new content must match the existing doc's format, naming conventions, and structure

## What it produces

1. **Updated doc files** — targeted section-level patches to existing docs
2. **Patch summary** — what was touched, what was skipped, and why, tied to the changes that drove each update
3. **`.doc-map.yaml`** — generated on first run if absent; maintained on subsequent runs. Maps doc paths → code paths they cover. Serves as KG seed.

## Doc scope

| Doc type | When patched | When skipped |
|----------|-------------|-------------|
| README.md | Public API or module-level changes | Internal refactors |
| Engineering guide | Almost always for behavioral changes | Cosmetic-only changes |
| Spec | Bug reveals missing requirement, or new requirement added | Refactors, internal changes |
| Spec-summary | Regenerated when spec changes | When spec is unchanged |
| Test-docs | Rarely — only when test structure changes | Most changes |
| Test-coverage.md | Incrementally when new tests are added | When no test changes |
| Human-facing config docs (.env.example) | When config options change | N/A for tool-consumed config |

**Out of scope:** tool-consumed config files (pyproject.toml, docker-compose.yaml, CI workflows). Detection heuristic: if the file is referenced in a build tool, CI pipeline, or package manager, it's code, not docs.

## Edge cases considered

| Edge case | Handling |
|-----------|---------|
| **No docs exist** | Delegate creation to the relevant write-* skill as a subagent, then patch the result. Only for docs directly relevant to the diff — no aspirational creation. |
| **Write-* skill not found** | Fail loudly: "Cannot find skill `write-engineering-guide`. Create the doc manually or check installed skills." |
| **Pre-existing staleness** | Ignore. Only address the diff. Staleness detection is a future KG/doc-audit concern — unreliable without structured code↔doc mapping. |
| **Cosmetic-only diff** | Exit cleanly with "no behavioral changes detected, nothing to patch." Threshold is behavioral impact, not diff size. |
| **Diff too large for patching** | If any doc would need a new top-level section, redirect to the specific write-* skill for that doc. |
| **Variable rename in public API** | Not cosmetic — update eng-guide and README if the name is user-facing. |

## Invocation modes

All three are valid — the skill must support each:

- **User-invoked:** `/patch-docs` after finishing a coding task
- **Pipeline stage:** registered in SKILLS_REGISTRY.yaml, fires after code-review
- **CLAUDE.md trigger:** lightweight rule "always run /patch-docs after any coding change" — logic stays in the skill, only the invocation trigger is in CLAUDE.md

For headless/autonomous execution: conservative defaults. Skip rather than guess on ambiguous changes.

## Companion files anticipated

- **references/doc-impact-map** — change-type → doc-type mapping rules. Which change types affect which doc types. Loaded during triage phase.
- **references/escalation-policy** — when to refuse and redirect to full regen. Loaded when patch-docs suspects a change is too large.
- **templates/patch-summary** — structured output format for the summary (what was patched, what was skipped, why).
- **examples/before-after** — worked example: small feature addition → targeted eng-guide section update showing what "consistent with existing conventions" looks like.

## Adjacent skills surfaced during brainstorm

- **write-test-coverage** — new skill to create and initialize test-coverage.md (the living test coverage register). Stub already created at `src/docs/write-test-coverage/SKILL.md`. Patch-docs updates this doc incrementally but does not create it.
- **doc-audit (future)** — staleness detection across all docs, powered by KG. Not patch-docs' job.

## Open questions

- **Config file detection heuristic** — "check if referenced in build/CI/package chain" is the candidate rule. Needs validation in practice — false negatives (missing a doc config) are acceptable, false positives (corrupting a build file) are not.
- **Change-type classification accuracy** — how well can the skill classify diffs into feature/refactor/bugfix from the diff alone? May need the commit message or PR description as supplementary input.
- **Pipeline registration** — what `input_type` and `output_type` for the SKILLS_REGISTRY.yaml entry? Candidate: `input_type: source_code | review_output`, `output_type: patched_docs`.
