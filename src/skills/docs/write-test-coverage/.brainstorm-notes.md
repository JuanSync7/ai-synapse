## Key Insights
- Stub already exists with clear intent from prior brainstorm (patch-docs session)
- Positioned as the missing layer between spec (what must be true) and test-docs (how to structure tests)
- Not about line coverage — about behavioral/scenario coverage
- Initialized once by this skill, maintained incrementally by /patch-docs
- **Dual audience**: humans scan it for gaps; agents read it to prioritize what to test next
- **Criticality ranking**: not all gaps are equal — Critical/High/Low per AC so agents pick highest-value work
- **Staleness**: generation-time git timestamp check, flag [stale?], verification is out of scope (separate agent/skill)
- **Cross-cutting section**: mirroring repo structure, cross-cutting lives at parent directory level, named by intersecting directories
- **Coverage status is derived, not judged**: each AC decomposes into scenario checklist, status = mechanical count (0/N = Not covered, partial = some, all = Covered)
- **Scenario source**: acceptance criteria from spec decompose into testable scenarios — this decomposition is the skill's core reasoning job
- **Language agnostic register, language-aware scanning**: register format is about behaviors not framework syntax; scanning needs per-language conventions (pytest, UVM, Jest, CI steps)
- **Intent docs as source of truth**: spec > eng-guide > docstrings/headers. NEVER infer "what should be tested" from code — that mirrors what exists and hides gaps
- **Non-testable artifacts**: infra files (Make, Docker, sh) — "coverage" means CI validation path, not test file
- **Loud failure on missing intent**: if no spec/eng-guide/docstrings exist for a module, flag it rather than guessing
- **Two-phase execution**: Phase 1 (bottom-up) = subagents scan test functions → behavioral summaries. Phase 2 (top-down) = single agent cross-matches summaries against ACs from intent docs
- **Orphaned tests**: tests that don't map to any AC are surfaced — might indicate incomplete spec or dead test code
- **Two-section output**: Test Inventory (what exists) + Coverage Gap Analysis (what's missing)
- **Fan-out mapping**: one AC can map to tests across multiple files — not a scaling problem, just a one-to-many match Phase 2 handles natively
- **No embeddings needed**: AC + summary counts fit in single context; chunk by module if scale becomes an issue
- **Four-section register structure**: (1) per-module sections mirroring repo, (2) cross-cutting section for multi-module ACs, (3) orphaned tests, (4) unfound modules (no intent docs)

## Lens Notes
### Usability
- Trigger phrases clear: "write test coverage", "coverage gaps", "what is tested"
- Name communicates intent
- Dual-audience output: humans scan visually, agents parse for next-action decisions
- Args: spec-path + test-dir, auto-discover if omitted

### Robustness
- Missing intent docs → loud failure, not silent guessing
- Staleness → git timestamp flag [stale?], no deep verification
- CI-only coverage → flagged as gap, not counted as coverage
- Large repos → chunk by module, no embeddings needed
- Phase 1 subagents are isolated — one failing doesn't block others

### Maintenance
- Language conventions in references/ file — easy to extend for new languages
- Register format is language-agnostic — survives stack changes
- Incrementally updated by /patch-docs — no full regeneration needed

### Preciseness
- Core injection: scenario decomposition from ACs (Claude doesn't do consistently without guidance)
- Classification policy: coverage status derived mechanically from checklist (not subjective judgment)
- Staleness flagging: cheap git timestamp check
- Language conventions: only what Claude can't derive from training (e.g., bats for shell, goss for Docker)

### Boundary
- /write-test-docs: implementation test plan (how to structure tests) — this skill is what to test, not how
- /write-module-tests: actual test code — this skill writes a document, not code
- /patch-docs: incremental updates — this skill initializes, patch-docs maintains
- /doc-authoring: pipeline router — this skill is a pipeline stage
