# Decision Memo — write-test-coverage

## What I want

A skill that generates a living test coverage register — a document that maps acceptance criteria from intent documents (spec, eng-guide) to actual test scenarios, tracking what's covered, what's partial, and what's missing. Serves two audiences: humans scanning for gaps and agents deciding what to test next.

## Why Claude needs it

Without this skill, Claude has four failure modes:

1. **Format drift** — Claude produces a coverage table but the shape varies across invocations. `/patch-docs` can't reliably parse and incrementally update an inconsistent format.
2. **No workflow awareness** — Claude doesn't know this document is initialized once and maintained incrementally by `/patch-docs`. Without that context, it regenerates the whole thing or produces something that breaks the maintenance contract.
3. **Inconsistent classification** — "Covered" vs "Partial" vs "Not Covered" is a judgment call without a policy. The skill makes it mechanical: each AC decomposes into a scenario checklist, status derives from checklist count (0/N = Not Covered, some = Partial, all = Covered).
4. **Shallow traceability** — Claude maps loosely ("this is probably tested") rather than tracing to specific test files, functions, and line numbers. The register demands specificity.

## Injection shape

**Formatting + policy.** The register template is the formatting injection (rigid structure for human scanning and `/patch-docs` parsing). The scenario decomposition logic and classification policy are the judgment injection (turning ACs into checklists, deriving status mechanically).

## What it produces

`test-coverage.md` — a register with four sections:

1. **Per-module sections** (mirroring repo structure) — ACs scoped to one module, with scenario checklists showing coverage status per scenario, linked to specific test files/functions
2. **Cross-cutting section** — ACs that span multiple modules (e.g., "all endpoints return structured errors"), with fan-out references across test files. Lives at the parent directory level, named by intersecting directories
3. **Orphaned tests** — test functions that don't map to any known AC. Signals incomplete spec or dead test code
4. **Unfound modules** — modules with no intent document (spec, eng-guide, or docstrings). Loud flag: "Cannot determine expected coverage — consider writing a spec or eng-guide first"

Each acceptance criterion entry looks like:

```
AC-3: Token refresh flow [MUST]
  [x] Happy path — refresh valid token          (test_auth.py:45)
  [ ] Expired refresh token returns 401
  [ ] Concurrent refresh race condition
  Status: Partial (1/3 scenarios)
```

## Execution model

**Two-phase, subagent-orchestrated internally:**

- **Phase 1 (bottom-up, parallel):** Subagent per test function/file. Each reads one test, returns a behavioral summary in structured format: `{test_file, test_function, module, behavior, type}`. No AC knowledge needed — keeps context tiny.
- **Phase 2 (top-down, single agent):** Loads all Phase 1 summaries + AC checklist from intent docs. Cross-matches summaries to ACs. Unmatched ACs = gaps. Unmatched tests = orphans. Produces the final register.

For small repos, both phases can run in a single pass without subagents.

## Key design decisions

- **Intent docs as source of truth for "what should be tested"**: spec > eng-guide > docstrings/headers. NEVER infer expected coverage from code itself — that mirrors what exists and hides gaps.
- **Language-agnostic register, language-aware scanning**: Register format is about behaviors, not framework syntax. A `references/language-conventions.md` file maps languages to test patterns (pytest, bats, goss, UVM, Jest, etc.).
- **CI is not a test**: If behavior is only validated by a CI step and has no dedicated test file, flag it as "CI-only validation — no dedicated test." Linters are static analysis, excluded from register.
- **Staleness flagging**: Generation-time git timestamp comparison (source file modified more recently than test file → `[stale?]` flag). Verification of whether a stale test is actually wrong is out of scope — that's a separate agent/skill concern.
- **Criticality lives in the AC**: Preserved from spec priority (MUST/SHOULD/MAY) or eng-guide emphasis. Skill doesn't invent its own ranking.
- **No embeddings needed**: AC + behavioral summary token counts fit single context. Chunk by module if scale becomes an issue.

## Edge cases considered

| Edge case | Handling |
|-----------|----------|
| No intent docs for a module | Loud failure: "No spec/eng-guide found — cannot determine expected coverage" |
| AC spans multiple modules (cross-cutting) | Cross-cutting section with fan-out references |
| Test exists but source code changed (stale) | `[stale?]` flag via git timestamp at generation time |
| Infra artifacts (Dockerfile, Makefile, shell) | Same framework — use proper test tools (bats, goss, container-structure-test) |
| Test doesn't map to any AC | Orphaned tests section |
| CI-only validation, no test file | Flagged as gap, not counted as coverage |
| Large repo with many modules | Chunk Phase 2 by module; Phase 1 subagents are already per-function |

## Companion files anticipated

- **references/language-conventions.md** — per-language test file patterns, frameworks, directory conventions (pytest, bats, goss, UVM, Jest, container-structure-test, etc.). Loaded during Phase 1 subagent prompt construction.
- **references/phase1-subagent-prompt.md** — template prompt for Phase 1 subagents: read test function, return structured behavioral summary. Loaded when dispatching subagents.
- **templates/test-coverage-register.md** — output template with all four sections, scenario checklist format, staleness flag syntax, cross-cutting entry format.
- **examples/example-register.md** — filled-in example showing all section types: per-module with checked/unchecked scenarios, cross-cutting with fan-out, orphaned tests, unfound modules, staleness flags.

## Open questions

- **Exact template verbosity** — how much detail per scenario line? The `(test_file.py:line)` reference is clear, but should it include test type (unit/integration/e2e) per line or per section?
- **Pipeline registration** — slots into `quality > testing` in SKILLS_REGISTRY.yaml. Input type likely `engineering_guide | formal_spec`. Output type `coverage_register`. Exact `requires_*` chain needs confirming — probably `requires_any: [eng-guide, spec]`.

## Registry placement

Under `quality > testing`, alongside `write-test-docs` and `write-module-tests`. No overlap with existing skills. Boundaries are clean:
- `write-test-coverage`: what is tested and what isn't (coverage register)
- `write-test-docs`: how to structure tests in code (test plan)
- `write-module-tests`: actual test code (pytest implementation)
